## 1. 总体技术决策

| 维度 | 选型 | 决策依据 |
| --- | --- | --- |
| 跨平台框架 | Flutter 3.22+（Dart 3.4+） | 性能接近原生满足 30Hz 仪表盘渲染；BLE 生态成熟；单人开发效率高 |
| 状态管理 | Riverpod 2.x | 编译期安全、可测、依赖注入清晰；适合数据源抽象层 |
| 路由 | go_router | 官方维护、深链友好 |
| 本地存储（敏感） | flutter_secure_storage | iOS Keychain / Android Keystore，存私钥 |
| 本地存储（一般） | Isar 4.x | 高性能、强类型、零样板 |
| BLE 通信 | flutter_reactive_ble | 流式 API 与 Dart 原生契合，跨平台稳定 |
| 加密原语 | pointycastle + cryptography | 完整 P-256 ECDSA、HKDF 支持 |
| Tesla 协议 | **Dart 重实现** | 详见 §3，避开 Go FFI 跨平台坑 |
| 序列化 | protobuf | Tesla 协议必需 |
| 测试 | flutter_test + mocktail + integration_test | 覆盖单元 / Widget / 集成 |
| CI | GitHub Actions（个人开发足够） | 免费额度够、配置简单 |
| 错误监控 | **占位**：Sentry 或 Bugly（W6 决策） | 单人需要崩溃可见性；最终方案需结合合规（境外/境内）评估，详见 §13 待决问题 |
| 包管理 / 构建 | Flutter 标准工具链 + Fastlane（提包） | Fastlane 简化 iOS / Android 多渠道发布 |

### 2.2 关键架构原则

1. **数据源抽象**：上层通过接口消费数据，不感知 GPS / CAN / BLE 差异
2. **单向数据流**：BLE/Sensor → Service → Notifier → UI，UI 事件 → Notifier → Service
3. **离线优先**：v1 不依赖任何网络服务（除崩溃上报），所有逻辑本地完成
4. **可测试**：Domain 层纯 Dart 无平台依赖，Infrastructure 层全部接口化便于 Mock

## 3. Tesla BLE Vehicle Command 集成方案

### 3.1 决策：Dart 重实现协议

**对比选项：**

| 方案 | 优势 | 劣势 | 决策 |
| --- | --- | --- | --- |
| Dart FFI 调用 Go SDK（gomobile） | 协议跟随官方 | iOS/Android FFI 配置复杂；Go 运行时膨胀 APK 8-15MB；调试困难 | ❌ |
| Dart 重实现 BLE 协议层 | 包体积可控；调试友好；跨平台一致 | 协议升级需手动跟进；初期需投入约 2 周阅读官方源码 | ✅ **采用** |
| Swift/Kotlin 原生模块 + MethodChannel | 性能好 | 维护两套；与 Flutter 选型矛盾 | ❌ |

**采用 Dart 重实现的可行性**：Tesla 官方协议文档 + Go 源码 + protobuf 定义是齐全的，Dart 的 `protobuf` 包和 `pointycastle` 完整覆盖所需加密原语。社区已有 Python / TS 第三方实现可作参考。

### 3.2 关键协议要点（实现层 checklist）

```
┌─ 配对流程 ─────────────────────────────────────────────┐
│ 1. App 生成 ECDSA P-256 密钥对                          │
│ 2. 私钥存入 flutter_secure_storage（iOS Keychain）     │
│ 3. 通过 BLE 发送 AddKey 请求 + 公钥                    │
│ 4. 用户在车机屏幕上确认（Tap card to add key）         │
│ 5. App 收到确认应答，存储 vehicle_key_id                │
└─────────────────────────────────────────────────────────┘

┌─ 命令签名流程（每次发送命令）──────────────────────────┐
│ 1. 构造 protobuf 消息（VCSEC / 车身命令）               │
│ 2. 与 vehicle_id + counter + epoch 拼接                 │
│ 3. 使用 ECDSA P-256 签名                                │
│ 4. BLE GATT 写入特征值                                  │
│ 5. 监听特征值通知，解析响应                             │
└─────────────────────────────────────────────────────────┘

┌─ 数据读取流程（车辆状态查询）──────────────────────────┐
│ 1. 发送 GetVehicleData 请求（指定字段子集）             │
│ 2. 接收 protobuf 响应                                   │
│ 3. 解码并发布到 VehicleStateStream                      │
└─────────────────────────────────────────────────────────┘
```

### 3.3 BLE 服务与特征（参考官方文档）

- **服务 UUID**：Tesla VCSEC 服务（按官方 SDK 定义）
- **写特征**：发送签名命令
- **通知特征**：接收响应
- **MTU**：协商至少 185 字节，分片处理大消息

### 3.4 风险点

- **配对体验**：用户需要把卡片或手机贴在中控台，引导文案必须清晰
- **私钥丢失场景与重配对流程**：用户卸载重装、清除数据、换手机、Keychain/Keystore 不可恢复时私钥都会丢失。详见 §3.5
- **多车主共享**：v1 不支持，v2 评估
- **协议变更**：必须订阅 Tesla GitHub repo 关注 release notes，详见 §3.6
- **开源协议合规**：Tesla `vehicle-command` 是 Apache 2.0，详见《03-合规专项》§15

### 3.5 私钥丢失与重配对设计

私钥一旦丢失无法恢复，必须重新配对。但 Tesla 车机会累积所有曾添加过的 key（最多 100 个）。设计重点：

**触发场景**

| 场景 | 私钥状态 | 处理 |
| --- | --- | --- |
| 用户卸载并重装 App | 丢失 | 全新配对流程 + 引导删除车机旧 key |
| 用户清除 App 数据 | 丢失 | 同上 |
| 用户换手机（未做迁移） | 丢失 | 同上 |
| 用户换手机（已做 Keychain iCloud 同步） | 保留 | 但出于安全策略我们禁止 iCloud 备份私钥（见 §9），实际仍丢失 |
| 私钥指纹不匹配（异常） | 视为丢失 | 强制重配对 |

**重配对 UX**

1. App 启动检测到本地无私钥但 Local DB 有车辆记录 → 进入"重新配对"流程
2. 步骤 1：明确提示"为了安全，原配对凭证已失效，需要重新配对"
3. 步骤 2：引导用户在车机上删除旧 key（图文步骤：进入车机"安全 → 钥匙"页面，删除"T-Dash"开头的卡片）
4. 步骤 3：常规配对流程
5. 配对成功后保留原车辆昵称等用户偏好，仅刷新 vehicle_key_id

**长期累积问题**

如果用户多次重配对，车机会累积多张同名 key。App 内提示"建议每次重新配对前先删除旧卡片，否则可能影响车主管理体验"。

### 3.6 协议变更监控与应急响应

**监控机制**

- 订阅 Tesla GitHub `vehicle-command` 仓库 Release Webhook（GitHub 免费功能）
- 每次 release notes 自动转发到开发者邮箱 + 可选企业微信
- 每月手动 review Tesla 车机软件更新公告

**应急响应级别**

| 严重度 | 触发 | 响应 SLA |
| --- | --- | --- |
| P0 | 协议不兼容，App 完全不可用 | 暂停发版；24 小时内排查定位；72 小时内热修复或紧急版本 |
| P1 | 部分命令失效 | 7 天内修复并发版；UI 内提示用户暂时不可用功能 |
| P2 | 新字段/新命令可选支持 | 纳入下个版本迭代 |

## 4. GPS 测速与传感器融合

### 4.1 目标

在常规驾驶场景（高速、市区主干道）下，显示车速与车机表读数偏差 ≤2 km/h 的概率 ≥90%。

### 4.2 数据源

| 来源 | 频率 | 优势 | 劣势 |
| --- | --- | --- | --- |
| GPS 速度（NMEA RMC / 平台 API） | 1Hz | 长期无漂移、绝对参考 | 隧道丢失、低速噪声大 |
| 加速度计 | 50-100Hz | 高频、不依赖卫星 | 积分漂移严重 |
| 陀螺仪 | 50-100Hz | 检测转向 | 积分漂移 |
| 气压计 | 选配 | 高度变化 | 与车速无关，可忽略 |

### 4.3 融合算法：扩展卡尔曼滤波（EKF）

状态向量：`[v, a]`（速度、加速度）

预测：`v_k = v_{k-1} + a_{k-1} * Δt`

**两阶段更新**：

1. **加速度计更新**（高频，50Hz）：用加速度计测量值更新加速度状态
2. **GPS 速度更新**（低频，1Hz）：用 GPS 速度修正速度状态，抑制积分漂移

**协方差设计**：GPS 测量噪声方差随 HDOP 动态调整；加速度计噪声方差固定。

### 4.4 边界处理

- **GPS 信号丢失**（HDOP > 5 或无定位）：仅依赖加速度计推算，超过 10 秒则 UI 显示"信号弱"占位
- **静止状态**：速度 < 0.5 m/s 时强制归零（避免 GPS 漂移导致原地"车速"跳动）
- **首次启动**：未取得 GPS 锁定前，UI 显示连接动画而非数字

### 4.5 数据源抽象设计

```dart
// Domain 层接口
abstract class VelocityProvider {
  Stream<VelocitySample> get velocityStream;
  VelocitySource get source;       // gps / canBus / fused
  ProviderHealth get health;       // healthy / degraded / unavailable
}

class VelocitySample {
  final double kmh;
  final DateTime timestamp;
  final double confidence;         // 0.0-1.0
  final VelocitySource source;
}

// Infrastructure 层实现
class GpsVelocityProvider implements VelocityProvider { ... }    // v1
class CanVelocityProvider implements VelocityProvider { ... }    // v2

// Application 层切换器
class VelocityProviderSelector {
  // 检测 OBD 适配器存在性，自动选择 Provider
  // UI 不感知差异
}
```

UI 层只消费 `VelocitySample.kmh` 和 `source`（用于诚实标注数据源）。

## 5. 数据源抽象层（核心架构原则）

### 5.1 全部需要抽象的数据维度

| 数据维度 | v1 来源 | v2 来源 |
| --- | --- | --- |
| 车速 | GPS Provider | CAN Provider |
| 电量 / 续航 | BLE | BLE（CAN 可补充瞬时功率） |
| 车门 / 锁状态 | BLE | BLE |
| 空调 | BLE | BLE |
| 充电状态 | BLE | BLE + CAN |
| 胎压 | BLE | BLE + CAN |
| 行车轨迹 | ❌ | CAN（v2 新增） |
| 能耗 | ❌ | CAN（v2 新增） |

### 5.2 Provider 注册与发现

启动时：

1. 必选：BLE Service 初始化
2. 必选：GPS Provider 初始化
3. 可选：扫描已知 OBD 适配器 BLE 广播
   - 若存在 → 注册 CanVelocityProvider 等高级 Provider，禁用 GpsVelocityProvider
   - 若不存在 → 仅使用 GPS

### 5.3 切换策略

- 冷启动时确定主 Provider，运行时不再频繁切换（避免数据跳变）
- OBD 适配器中途断开 → 显示提示，下次启动重新评估
- 用户可在设置中强制使用某个 Provider（调试/降级）

## 9. 安全设计

| 项 | 设计 |
| --- | --- |
| 私钥存储 | iOS Keychain / Android Keystore，禁止备份到 iCloud |
| 命令完整性 | 每条命令含 counter + 时间戳，防重放 |
| 本地数据库加密 | Isar 启用 AES 加密，密钥存 Keystore |
| 网络流量 | v1 仅崩溃监控 SDK 上报（Sentry / Bugly），HTTPS + 证书锁定 |
| 反逆向 | Flutter --obfuscate + --split-debug-info |
| 隐私 | 不收集 PII；GPS 数据不离开设备 |
